# Three-Layer Architecture

Karpathy's LLM Wiki system uses three distinct layers with clear ownership, purpose, and access patterns.

## Layer 1: raw/ - Source of Truth

**Definition:** The collection of source files you choose. Articles, papers, images, data files. These are immutable—LLM only reads them, never writes. This is your source of truth.

**Access pattern:** Read-only for LLM, append-only for humans

**Example structure:**
```
raw/
  articles/
    2026-03-attention-is-all-you-need-revisited.md
    2026-04-scaling-laws-update.md
  papers/
    transformer-architecture-v2.pdf
    mixture-of-experts-survey.pdf
  repos/
    llama-3-readme.md
    vllm-architecture-notes.md
  data/
    benchmark-results-model-comparison.json
  images/
    transformer-diagram.png
    scaling-curves.png
  assets/
    # Downloaded images from clipped articles
```

**Why immutable:** You always have original source for re-reading. If LLM makes error in wiki, you can return to source and correct.

## Layer 2: wiki/ - LLM-Generated Knowledge

**Definition:** Directory of Markdown files generated by LLM. Includes summaries, entity pages, concept pages, comparisons, overviews, and synthesis. LLM has complete control. Creates pages, updates on new resources, maintains cross-references, ensures consistency. You read it, LLM generates it.

**Access pattern:** Read for humans, read/write for LLM

**Example structure:**
```
wiki/
  index.md              # Master directory of all pages
  log.md                # Chronological activity log
  overview.md           # High-level synthesis
  concepts/
    attention-mechanism.md
    mixture-of-experts.md
    scaling-laws.md
    tokenization.md
  entities/
    openai.md
    anthropic.md
    google-deepmind.md
  sources/
    summary-attention-revisited.md
    summary-scaling-update.md
  comparisons/
    gpt4-vs-claude-vs-gemini.md
    rag-vs-finetuning.md
```

**Key insight:** Wiki sits between you and raw data. You don't read raw papers to answer questions—you read wiki. Wiki is pre-digested, cross-referenced, synthesized.

## Layer 3: Schema - The Configuration

**Definition:** Document (e CLAUDE.md for Claude Code, AGENTS.md for Codex) that tells LLM wiki's structure, conventions, and workflow for ingesting sources, answering questions, or maintaining wiki. This is the critical configuration file—it makes LLM a disciplined wiki maintainer, not just a generic chatbot.

**Example content:**
```markdown
# LLM Wiki Schema

## Project Structure
- `raw/` — Immutable source documents. Never touch.
- `wiki/` — LLM-generated wiki. You own this documentation.
- `wiki/index.md` — Master table of contents. Update on every ingest.
- `wiki/log.md` — Append-only log of activities.

## Page Specification
Every wiki page must have YAML frontmatter:
---
title: Page Title
type: concept | entity | source summary | comparison
sources: [list of cited raw/ files]
related: [list of linked wiki pages]
created: YYYY-MM-DD
updated: YYYY-MM-DD
confidence: high | medium | low
---

## Ingest Workflow
When I say "ingest [filename]":
1. Read source file in raw/
2. Discuss key points with me
3. Create/update summary page in wiki/sources/
4. Update wiki/index.md
5. Update all related concept and entity pages
6. Append entry to wiki/log.md

## Query Workflow
When I ask questions:
1. Read wiki/index.md to find relevant pages
2. Read those pages
3. Synthesize answer with [[wiki-link]] references
4. If answer valuable, suggest archiving as new wiki page
```

**Why schema matters:** Without schema, each session with LLM starts from zero. LLM doesn't know your conventions, page formats, or workflows. Schema is persistent memory—carries knowledge across sessions, ensures consistency. Generic LLM becomes **your** wiki maintainer.

**Evolution:** Schema is not static. You refine it as you discover what works for your domain. Start from basic version, iterate as pages reveal better structures, frontmatter fields, workflows.

# 第 1 章：提示词链（Prompt Chaining）

这是《Agentic Design Patterns（智能体设计模式）》一书的第一章，介绍了**提示词链**模式——这是构建复杂 AI 智能体系统的基础技术。

## 概述

提示词链（也称为管道模式 Pipeline Pattern）是分而治之的策略：将复杂问题分解为一系列更小、更易管理的子问题，每个子问题通过专门设计的提示词单独处理，一个提示词的输出会作为输入策略性地传递给链中的下一个提示词。

这种方法解决了**单一提示词**处理复杂任务时的多个问题：
- 指令忽略：部分提示内容被忽视
- 上下文偏离：模型失去对初始上下文的追踪
- 错误传播：早期错误被放大
- 上下文窗口不足
- 幻觉增加：认知负荷增加导致生成错误信息

## 核心优势

1. **模块化与可调试性**：每个步骤专注于特定方面，更容易理解和调试
2. **清晰的依赖链**：前一步的输出作为后一步的输入，逐步构建解决方案
3. **支持外部工具集成**：每一步都可以与外部系统、API 或数据库交互
4. **可解释性增强**：流程透明，每个步骤可以单独优化
5. **是构建复杂智能体系统的基础**：支持多步推理、规划和决策

## 关键实践：结构化输出

提示词链的可靠性高度依赖于步骤之间传递的数据完整性。为了缓解歧义问题，应指定结构化输出格式（如 JSON 或 XML）。结构化格式确保数据是机器可读的，可以精确解析并无歧义地插入到下一个提示词中。

示例：
```json
{
  "trends": [
    {
      "trend_name": "AI-Powered Personalization",
      "supporting_data": "73% of consumers prefer..."
    }
  ]
}
```

## 主要应用场景

| 应用场景 | 说明 |
|---------|------|
| **信息处理工作流** | 提取文本 → 总结 → 提取实体 → 搜索知识库 → 生成报告 |
| **复杂查询回答** | 分解子问题 → 分别研究 → 综合答案 |
| **数据提取和转换** | 尝试提取 → 验证 → 条件性修正 → 迭代直至正确 |
| **内容生成工作流** | 构思 → 大纲 → 分段起草 → 审查完善 |
| **有状态对话智能体** | 识别意图 → 更新对话状态 → 基于状态生成响应 |
| **代码生成和完善** | 理解需求 → 生成大纲 → 编写代码 → 审查改进 → 添加文档 |
| **多模态多步推理** | 分解不同模态任务，逐步处理整合 |

## 上下文工程

本章介绍了**上下文工程**的概念——这是从传统提示工程演进而来的方法论：
- **传统提示工程**：主要专注于优化单个查询的措辞
- **上下文工程**：系统地设计、构建和向 AI 模型提供完整信息环境，包括：系统提示词、检索文档、工具输出、隐式数据（用户身份、交互历史、环境状态）

上下文工程认为模型输出的质量较少依赖于模型架构本身，更多依赖于所提供上下文的丰富性。这是将无状态聊天机器人提升为情境感知智能体系统的关键方法论。

## 代码示例

本章提供了一个完整的 Python 示例，使用 **LangChain LCEL**（LangChain Expression Language）实现两步提示词链：
1. 第一步：从非结构化文本中提取技术规格
2. 第二步：将提取的规格转换为结构化 JSON

示例使用了笔记本电脑描述作为输入，演示了整个处理流程。

## 关键要点

- 提示词链将复杂任务分解为一系列较小的、聚焦的步骤
- 链中的每一步都使用前一步的输出作为输入
- 这种模式提高了可靠性和可管理性
- 框架如 [[concepts/langchain|LangChain]]、[[concepts/langgraph|LangGraph]]、Google ADK、Crew AI 提供了健壮的工具来定义和执行这些多步序列

## 经验法则

当满足以下条件时使用提示词链模式：
- 任务对于单个提示词过于复杂
- 涉及多个不同的处理阶段
- 需要在步骤之间与外部工具交互
- 正在构建需要执行多步推理并维护状态的智能体系统
